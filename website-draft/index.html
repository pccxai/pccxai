Contribute

The best first contribution is a clear one.

Start with docs, tests, issue cleanup, examples, or small validation tasks. Keep claims specific. Attach evidence. Ask early.

Open contribution guide
Choose the right repo.Docs, RTL, lab tooling, launcher contracts, IDE surfaces, or staging.
Keep PRs narrow.One change, one reason, one validation path.
Do not overclaim.Measured, simulated, planned, and target results must stay separate.

Assistant preview

A quiet help surface, not a model demo.

The current assistant page is static. It does not call a model service, does not use an API key, and cannot spend credits. Live model access comes later, behind protection.

Question
How do I start contributing?
Answer
Read the contribution guide, choose one repository, open or comment on an issue, then submit a small PR with validation notes.

Security

Static first. Runtime later.

The site is static today. No client-side model key. No paid API endpoint. Future assistant work will require Turnstile, server-side verification, rate limits, provider budget guards, and logging.

0

No paid endpoint

The preview cannot burn model credits because there is no runtime provider call.

T

Turnstile later

When chat becomes live, requests must pass server-side Turnstile validation.

RL

Rate limits

Public assistant traffic needs per-IP, per-session, and daily budget limits.
